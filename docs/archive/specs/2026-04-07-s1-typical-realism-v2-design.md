# S1 Typical Realism V2 Design

> Status: approved direction after rejecting optimizer-side family-window constraints as too invasive for the current paper-facing benchmark.

## 1. Goal

Improve `s1_typical` realism again while keeping the benchmark clean:

- make generated layouts visibly tighter
- make baseline and optimized temperature span larger
- keep `raw` and `union` comparable without adding optimizer-specific spatial policy
- keep the active `32D`, `x/y`-only, single-sink, single-case paper-facing contract

This revision intentionally does **not** add:

- family-aware optimizer windows
- zone-restore logic in optimizer repair
- new paper-facing objectives or constraints
- benchmark-specific optimizer policy hacks

## 2. Problem Statement

Recent work improved realism substantially, but three issues remain.

### 2.1 Gradient Page Is Only Qualitatively Correct

The optimization objective `summary.temperature_gradient_rms` is physically grounded because it is assembled directly from the finite-element solution as:

- `sqrt((1 / |Omega|) * integral |grad(T)|^2 dx)`

However, the rendered gradient field page is currently generated by:

1. sampling temperature DOFs to a regular grid with nearest-neighbor assignment
2. computing `np.gradient(...)` on that raster

This is acceptable as a visual diagnostic but it exaggerates local peaks and blockiness near sink edges and source boundaries.

### 2.2 Temperature Contrast Is Still Too Small

The active template already moved from the earlier `~2 K` regime to a `~9-10 K` baseline regime, but the real `raw/union` optimization runs observed representative spans closer to `~5.6-6.9 K`.

This remaining flatness comes from:

- moderate conductivities still smoothing heat aggressively
- localized sources that are present but not yet concentrated enough
- objective pressure from `temperature_max` plus `temperature_gradient_rms`, which naturally encourages flatter fields

### 2.3 Layout Still Reads Too Loose

The current dense-core logic is better than the original sparse baseline, but the deck still looks roomy because:

- `clearance` metadata exists but is not enforced in legality checks
- compactness refinement still optimizes mostly for bbox size plus mild zone/group penalties
- the dense-core zone is still broader than needed
- total component footprint on the panel is still materially below the reference legacy dataset's `0.4-0.5` area-ratio regime

## 3. Rejected Direction

The following direction was considered and explicitly rejected:

- restrict optimizer design-variable ranges with family-aware windows
- add optimizer-side zone restore in repair
- inject layout realism into optimizer mechanics

Why it was rejected:

- increases benchmark-specific machinery
- makes `raw/union` less clean as algorithm baselines
- complicates repair and search behavior
- risks turning optimizer policy into hidden spatial prior enforcement

Realism should first be improved in:

- template geometry
- generator legality and compactness
- solver/source calibration
- visualization/reporting correctness

## 4. Selected Design

### 4.1 Keep Optimizer Semantics Clean

Do not change:

- `scenarios/optimization/s1_typical_raw.yaml`
- `scenarios/optimization/s1_typical_union.yaml`
- optimizer objectives
- optimizer constraints
- decision encoding

The optimizer may still spread layouts if that improves objectives. That is acceptable for now. This phase focuses on making the base geometry and thermal model more realistic without encoding extra optimizer policy.

### 4.2 Make `clearance` Real

The generator already carries family-level `clearance`, but current legality only rejects direct polygon overlap.

Revise layout legality so generation-time placement and compactness refinement treat each footprint as if buffered by its required clearance before overlap testing.

Intent:

- preserve believable assembly gaps
- remove fake compactness created by near-touching components
- make compactness scoring meaningful relative to actual manufacturable spacing

This change is generator-only and does not alter paper-facing optimizer variables.

### 4.3 Tighten Compactness Scoring With Void Awareness

Do not add `panel_fill_ratio` to runtime layout penalty because component area is fixed under the current benchmark and therefore not discriminative.

Instead, strengthen the generation-time layout score with terms that actually vary under fixed geometry:

- bbox envelope compactness
- nearest-neighbor gap tendency
- adjacency-group dispersion
- dense-core large-void penalty

The new void term should detect when the dense-core zone contains a large contiguous empty patch even though components are technically legal.

Recommended implementation:

- evaluate a coarse occupancy grid inside `dense_core`
- compute the largest connected empty patch ratio
- penalize large empty patches

This stays generic, geometry-based, and benchmark-compatible.

### 4.4 Narrow Generation Zones Further

Keep generation-only zones, but shrink them moderately:

- tighten `dense_core`
- tighten `top_sink_band`
- keep edge lanes recognizable

This should push the generated baseline toward a more credible electronics-deck silhouette without adding optimizer-side region locks.

### 4.5 Thermal Realism V2

Increase thermal contrast through the existing physics path rather than adding new equation classes.

Changes:

- reduce conductivities another step from `5 / 20` toward a moderate lower-conductivity regime
- further localize high-power source support via smaller `source_area_ratio`
- optionally rebalance family powers into clearer hot / warm / cool tiers if needed after calibration
- keep the single top-edge sink model

This should increase baseline span while preserving the current active architecture.

### 4.6 Render Gradient Field From FE Gradient, Not Raster Differences

Keep `temperature_gradient_rms` exactly as-is.

But revise page/export diagnostics so `gradient_magnitude_grid` is sampled from the finite-element gradient field rather than reconstructed from a nearest-neighbor temperature raster.

Goals:

- visually smoother and more truthful diagnostic fields
- avoid artificial gradient spikes caused by rasterization
- align rendered hotspot/edge interpretation better with the PDE solution

### 4.7 Recompute Layout Metrics For Representative Bundles

Current representative bundle pages can show stale `layout_metrics` inherited from earlier provenance rather than recomputed metrics for the optimized representative case.

Fix the representative export path so the bundle layout diagnostics are recomputed from the representative `case.yaml` before page rendering and comparison summaries.

This is necessary for trustworthy inspection of compactness after optimization.

## 5. Scope

### In Scope

- `clearance` enforcement in generation legality
- stronger layout compactness scoring
- narrower generation zones
- template geometry/thermal retune
- FE-gradient-based diagnostic export
- representative layout metric recomputation
- tests, docs, and real validation reruns

### Out Of Scope

- optimizer family windows
- optimizer-side zone restore
- new optimization objectives
- new optimization constraints
- reviving `radiation_gen` runtime architecture

## 6. Validation Targets

The updated mainline should be accepted only if all of the following hold.

### Layout

- calibration seeds still generate without component dropouts
- `clearance` is enforced in tests
- baseline layouts show lower large-void tendency and stronger compactness
- representative bundle layout metrics are recomputed from actual optimized cases

### Thermal

- baseline spans move upward from the current `~9-10 K` regime toward approximately `12-18 K`
- rendered gradient fields no longer exhibit obvious raster stair-step artifacts
- optimization representatives keep physically interpretable hotspot / sink gradient structure

### Architecture

- `raw` and `union` specs remain structurally unchanged
- no new benchmark-specific optimizer mechanisms
- no contract regression for `s1_typical`
